# 摘要

随着程序设计类课程教学规模的扩大，传统人工批改作业方式在效率、反馈速度和评分一致性方面逐渐暴露出不足。教师需要投入大量时间检查代码运行结果，学生也难以及时获得针对性的评测反馈。针对上述问题，本文设计并实现了一套面向程序设计课程的作业自动评测与管理系统。系统基于 Spring Boot 框架开发，采用 Spring MVC 提供 RESTful 接口，使用 Spring Data JPA 完成数据持久化，支持 MySQL 正式环境和 H2 演示测试环境。系统面向管理员、教师和学生三类角色，主要功能包括用户认证、课程管理、学生选课、作业发布、测试用例配置、代码提交、自动评测、结果查询、成绩统计与操作日志等。

在自动评测方面，系统通过 Java 编译器对学生提交的源码进行编译，并在受控流程中运行程序，将实际输出与教师设置的期望输出进行比对，从而生成评测状态、分数和测试用例明细。系统能够识别通过、部分通过、错误答案、编译错误、运行错误和超时等结果。测试结果表明，系统能够完成从教师发布作业到学生提交代码、自动判题、教师查看结果的完整业务闭环，具备较好的教学辅助价值和毕业设计展示价值。

关键词：Spring Boot；自动评测；作业管理；Java 编译；程序设计教学

# ABSTRACT

With the expansion of programming courses, traditional manual assignment grading has gradually shown limitations in efficiency, feedback timeliness and scoring consistency. Teachers need to spend a large amount of time checking program outputs, while students cannot always receive immediate feedback on their submissions. To address these problems, this thesis designs and implements an assignment auto-grading and management system for programming courses. The system is developed based on Spring Boot, provides RESTful APIs through Spring MVC, persists data with Spring Data JPA, and supports MySQL for normal deployment as well as H2 for demonstration and testing. The system supports three roles, including administrator, teacher and student. Its main functions include user authentication, course management, course enrollment, assignment publishing, test case configuration, code submission, automatic judging, result query, grade statistics and audit logs.

For the auto-grading module, the system compiles submitted Java source code with the Java compiler, executes the program in a controlled process, compares the actual output with the expected output configured by teachers, and then generates judging status, score and detailed test case results. The system can handle accepted submissions, partial accepted submissions, wrong answers, compilation errors, runtime errors and time limit exceeded cases. Test results show that the system can complete the whole business process from assignment publishing to student submission, automatic grading and result review, which proves that it has practical value for teaching assistance and graduation project demonstration.

KEY WORDS: Spring Boot; Auto Grading; Assignment Management; Java Compilation; Programming Education

# 1 绪论

## 1.1 研究背景

程序设计课程是软件工程、计算机科学与技术、物联网工程等专业的重要基础课程。该类课程具有实践性强、作业频率高、代码结果可验证等特点。学生需要通过反复编写程序、调试程序和分析错误来形成编程能力，教师则需要通过作业批改和实验评价了解学生对知识点的掌握情况。然而，在实际教学过程中，人工批改程序设计作业往往存在工作量大、反馈周期长和评分标准不稳定等问题。当一个班级有数十名甚至上百名学生时，教师需要逐份下载或查看代码，再手工运行程序、构造输入、比较输出并记录成绩，这不仅耗费时间，也容易因为测试数据不统一而影响评分公平性[1]。

随着高校信息化建设的推进，教学管理系统、在线学习平台和在线评测系统被广泛应用于课程教学。在线评测系统通常能够根据预设测试用例自动运行学生程序，给出运行结果和分数，适合程序设计类课程的过程化评价[2]。但是，常见的在线评测平台多面向算法训练和程序设计竞赛，侧重题库、排名和高并发评测，而本科毕业设计中的课程管理场景还需要关注教师发布作业、学生选课、教学班管理、提交记录追踪和成绩统计等业务。因此，设计一个贴合程序设计课程教学管理流程的作业自动评测系统，具有较强的现实意义。

本课题选择“作业自动评测与管理系统”作为研究对象，主要原因在于该系统能够把课程管理、作业发布、学生提交和自动判题结合起来，形成一个相对完整的教学闭环。系统并不追求构建大型在线竞赛平台，而是面向本科教学中的常见需求，重点解决作业发布后学生如何提交、系统如何自动评测、教师如何查看结果、管理员如何管理基础数据等问题。通过该课题的设计与实现，可以综合运用 Web 后端开发、数据库设计、权限控制、前端交互、Java 程序编译运行和自动化测试等知识，符合软件工程专业毕业设计对系统性和实践性的要求。

## 1.2 研究意义

本系统的研究意义主要体现在教学管理、学生学习和工程实践三个方面。首先，从教学管理角度看，系统能够将重复性的代码运行与输出比对工作交给程序自动完成，使教师从大量机械性批改中解放出来，把更多精力投入到教学设计、难点讲解和个性化辅导中。教师可以通过统一的测试用例评价学生提交结果，减少人工判断差异，提高评分的一致性和可追溯性[3]。

其次，从学生学习角度看，及时反馈是程序设计学习中十分关键的环节。学生提交代码后，如果能够快速得知编译是否通过、运行结果是否正确、哪些测试用例失败，就可以更有针对性地修改程序。相比传统课后等待教师批改的方式，自动评测系统能够缩短反馈周期，有助于学生形成“编写—提交—反馈—修正”的学习循环。对于基础薄弱的学生，编译错误和运行错误提示也能帮助其发现常见问题。

再次，从工程实践角度看，本课题能够训练完整的软件系统开发能力。系统不仅包括登录注册、权限控制、数据增删改查等常规 Web 功能，还包括 Java 自动评测这一具有一定复杂度的业务模块。该模块涉及源码长度校验、类名识别、编译诊断、进程执行、超时控制、输出截断和结果计算等问题，能够体现毕业设计的技术含量。同时，项目通过 Maven 自动化测试验证关键流程，使系统不仅“能运行”，也具有一定的可维护性和可验证性。

## 1.3 国内外研究现状

在线评测系统的思想最早广泛应用于程序设计竞赛和算法训练平台。国外较有代表性的平台包括 UVa Online Judge、HackerRank、LeetCode 等，这类系统能够根据输入输出数据自动判断程序正确性，并为用户提供刷题、提交记录和排行榜等功能[4]。国内也有洛谷、牛客、在线课程平台中的编程题评测模块等应用。这些系统推动了程序设计学习方式的变化，使学习者能够在大量练习中不断获得反馈。

在高校教学场景中，自动评测系统逐渐从竞赛平台扩展到课程作业管理。部分学校搭建了面向本校课程的 OJ 平台，教师可以发布实验题目，学生在线提交代码，系统自动评测并统计成绩。这类平台在减轻教师负担方面具有明显优势，但也存在一些问题。例如，有些系统功能重心偏向题库和排名，对教学班级、课程选课、教师作业管理和成绩导出支持不足；有些系统部署复杂，对本科毕业设计或小规模课程使用来说成本较高；还有些系统界面和权限管理较弱，难以支撑管理员、教师、学生三类角色的完整演示[5]。

从技术实现角度看，自动评测通常需要解决程序编译运行、资源限制、安全隔离和结果判定等问题。成熟平台一般会采用容器、虚拟机或沙箱技术对用户程序进行隔离，以防止恶意代码访问宿主机资源[6]。对于本科毕业设计项目而言，如果直接实现高强度沙箱和分布式评测平台，难度和工作量都会较大。因此，本系统结合教学演示场景和项目规模，采用较轻量的 Java 编译运行机制，并通过源码关键字限制、执行超时控制、输出长度限制等方式降低风险。论文中也会如实说明该方式适合教学演示和课程辅助场景，不直接等同于生产级在线评测沙箱。

# 2 相关技术介绍

## 2.1 Spring Boot 技术

Spring Boot 是基于 Spring 生态的快速开发框架，它通过自动配置、起步依赖和内嵌服务器等机制降低了 Java Web 项目的搭建成本[7]。传统 Spring 项目往往需要编写大量 XML 或 Java 配置，而 Spring Boot 可以根据项目依赖自动推断并创建常用组件，使开发者将更多精力放在业务实现上。本系统采用 Spring Boot 作为后端基础框架，主要原因在于它适合构建单体 Web 应用，便于快速集成 Spring MVC、Spring Data JPA、校验、日志和测试模块。

在本系统中，Spring Boot 负责应用启动、依赖注入、配置管理和 Web 服务运行。项目入口类为 `AssignmentAutoGradingApplication`，通过 `SpringApplication.run` 启动整个系统。系统使用 `application.yml` 管理数据库连接、端口、编码、判题超时和日志级别等配置，并通过 `application-demo.yml` 提供 H2 内存数据库演示环境。这样的配置方式使系统能够在正式环境连接 MySQL，在演示环境快速启动，不需要额外安装数据库即可完成答辩展示。

## 2.2 Spring MVC 与 RESTful 接口

Spring MVC 是 Spring Web 模块中的核心组件，用于处理 HTTP 请求并返回响应。在本系统中，后端通过 Controller 层对外提供 RESTful 接口，前端页面通过 JavaScript 调用这些接口完成登录、查询、创建、提交和统计等操作。RESTful 风格强调资源和 HTTP 方法的对应关系，例如使用 `GET` 查询资源，使用 `POST` 创建资源，使用 `PUT` 修改资源，使用 `DELETE` 删除资源[8]。

系统中的主要接口包括 `/api/auth`、`/api/users`、`/api/courses`、`/api/assignments`、`/api/submissions`、`/api/import`、`/api/admin` 和 `/api/audit-logs`。例如，教师创建作业使用 `POST /api/assignments`，学生提交代码使用 `POST /api/submissions`，教师查看某作业提交记录使用 `GET /api/submissions/assignment/{assignmentId}`。这种接口设计清晰地反映了系统资源结构，有利于前后端分离和后续功能扩展。

## 2.3 Spring Data JPA

Spring Data JPA 是 Spring 对 JPA 数据访问技术的进一步封装，能够通过 Repository 接口简化数据库操作[9]。开发者只需要定义实体类和 Repository 接口，就可以获得常见的增删改查方法，也可以通过方法命名或注解查询实现复杂检索。本系统使用 JPA 将 Java 实体类映射到数据库表，将用户、课程、作业、提交等业务概念转化为持久化对象。

例如，`User` 实体对应用户表，`Course` 实体对应课程表，`Assignment` 实体对应作业表，`Submission` 实体对应提交记录表。系统通过 `UserRepository`、`CourseRepository`、`AssignmentRepository` 和 `SubmissionRepository` 等接口访问数据库。对于需要关联查询的场景，例如教师查看作业时同时加载测试用例和课程信息，系统通过实体图或自定义查询减少懒加载带来的问题。使用 Spring Data JPA 的优势是能够减少模板代码，提高开发效率，同时保持面向对象的数据建模方式。

## 2.4 MySQL 与 H2 数据库

MySQL 是常用的关系型数据库管理系统，具有成熟稳定、社区活跃、工具丰富等特点，适合保存系统的长期业务数据[10]。本系统默认配置连接 MySQL，适合正式运行和部署。数据库中保存用户、课程、作业、提交、评测结果、登录令牌和操作日志等数据。通过 Flyway 迁移脚本，系统可以在 MySQL 环境中创建并维护表结构。

H2 是一个轻量级 Java 数据库，支持内存模式和 MySQL 兼容模式，适合本地演示和自动化测试[11]。本系统在 `demo` profile 中使用 H2 内存数据库，并自动创建演示账号、课程、选课关系和作业数据。这样做可以避免答辩或本地测试时依赖外部 MySQL 环境，提高系统启动的便利性和可复现性。教师或评阅人只需运行 demo 启动命令，就可以直接使用 `admin1`、`teacher1`、`student1` 三个账号演示主要流程。

## 2.5 Maven 项目构建工具

Maven 是 Java 项目中常用的依赖管理和构建工具。它通过 `pom.xml` 描述项目依赖、插件、构建流程和项目元信息[12]。本系统使用 Maven 管理 Spring Boot、Spring Web、Spring Data JPA、数据库驱动、Flyway 和测试框架等依赖。开发过程中可以通过 `mvn test` 执行自动化测试，也可以通过 `mvn spring-boot:run` 启动系统。

Maven 的统一构建方式对毕业设计项目具有重要意义。一方面，它使项目在不同机器上更容易复现，评阅人只需要安装合适版本的 JDK 和 Maven，即可按照 README 中的命令运行系统。另一方面，Maven 测试阶段能够自动执行 JUnit 测试，帮助开发者及时发现回归问题。本系统在最终验证时执行了 35 个自动化测试用例，构建结果为 `BUILD SUCCESS`，说明 Maven 构建流程已经覆盖核心业务验证。

## 2.6 Java 自动评测技术

自动评测模块是本系统区别于普通教学管理系统的关键部分。对于 Java 程序，自动评测通常包括源码接收、类名识别、编译、运行、输入输出处理、超时控制、结果比对和分数计算等步骤[13]。本系统使用 JDK 提供的 `JavaCompiler` 对学生提交的源码进行编译。如果编译失败，系统保存编译错误信息并返回 `COMPILE_ERROR` 状态；如果编译成功，系统根据作业测试用例依次运行程序，将输入数据写入进程标准输入，再读取标准输出和错误输出。

为了适应教学演示场景，系统对提交源码长度、输出长度和执行时间进行了限制。若程序运行超过配置时间，则返回超时状态；若程序抛出异常或非正常退出，则返回运行错误；若输出与期望输出完全匹配，则该测试用例通过。最终系统根据通过用例数量计算得分，并生成 `ACCEPTED`、`PARTIAL_ACCEPTED`、`FAILED`、`RUNTIME_ERROR` 或 `TIME_LIMIT_EXCEEDED` 等状态。该方案虽然不是生产级沙箱，但已经能够满足本科课程作业自动评测和毕业设计展示的核心需求。

# 3 系统需求分析

## 3.1 系统总体需求

本系统面向程序设计课程中的作业管理与自动评测场景，目标是为教师、学生和管理员提供一个统一的平台。系统应支持教师创建课程、管理学生选课、发布作业、配置测试用例；支持学生查看已选课程和已发布作业，在线提交 Java 代码并查看评测结果；支持管理员查看用户、课程、作业和操作日志，维护系统基础数据。系统还应提供成绩统计和导出能力，帮助教师掌握作业完成情况。

从业务闭环角度看，系统的核心流程是：教师创建课程并将学生加入课程；教师在课程下发布作业并配置输入输出测试用例；学生登录后看到自己已选课程和课程下已发布作业；学生提交 Java 源码；系统自动编译运行并生成评测结果；学生查看自己的提交摘要和详情；教师查看该作业下所有学生提交记录和统计结果。该流程覆盖了程序设计课程作业管理的主要环节，也是系统设计和测试的重点。

## 3.2 用户角色分析

系统设置管理员、教师和学生三类角色。管理员主要负责全局数据查看和基础管理，包括查看用户列表、课程列表、作业列表和操作日志。管理员角色的存在有助于系统整体维护和演示管理，也便于在答辩中展示系统权限层次。

教师是系统中的主要业务管理者。教师可以创建课程、修改课程、管理课程中的学生选课关系；可以创建作业、设置作业状态、截止时间、评分策略和最大提交次数；可以为作业配置测试用例；可以查看学生提交记录、提交详情、评测结果和成绩统计。教师角色对应真实教学中的任课教师，是作业发布和评测规则制定者。

学生是系统中的提交者和反馈接收者。学生登录后只能看到自己已选课程和对应课程下已发布的作业，不能访问教师的管理接口，也不能查看其他学生的提交记录。学生可以选择作业提交代码，系统评测完成后可以查看提交状态、分数、编译信息、运行信息和测试用例结果。通过角色隔离，系统能够保证教学数据的基本安全性和可控性。

## 3.3 功能需求分析

管理员功能需求包括用户查询、课程查询、作业查询、操作日志查询和分页搜索等。管理员可以从系统层面查看整体数据，但不直接参与学生提交过程。管理员功能为系统维护提供支持，也方便答辩时展示后台管理能力。

教师功能需求较为复杂，主要包括课程管理、选课管理、作业管理、测试用例管理、提交查看和统计导出。课程管理要求教师能够创建课程，设置课程代码、课程名称、学期和班级；选课管理要求教师能够将学生加入课程或从课程中移出；作业管理要求教师能够发布作业，设置标题、描述、截止时间、状态、最大提交次数、是否允许迟交和评分策略；测试用例管理要求教师能够为作业设置多组输入和期望输出；提交查看要求教师能够查看自己作业下的学生提交，并可查看源码、编译结果、运行结果和测试用例明细。

学生功能需求包括查看已选课程、查看已发布作业、提交代码、查看最新提交和查看评测详情。学生端最重要的是数据范围控制：学生只能看到自己已加入课程下的已发布作业，不能看到草稿作业和关闭作业，也不能提交未选课程下的作业。这样可以保证系统符合课程教学管理逻辑。

## 3.4 非功能需求分析

在安全性方面，系统应对未登录请求和越权请求进行拦截。未携带有效 token 的请求应返回未认证错误，角色权限不足的请求应返回禁止访问错误。学生不能访问教师管理接口，教师不能管理其他教师创建的课程和作业。自动评测模块应限制源码长度、执行时间和输出长度，降低异常程序对系统运行的影响。

在可维护性方面，系统应采用分层架构，将控制器、业务服务、数据访问、实体模型和评测逻辑分离。不同模块之间通过清晰的接口交互，避免业务逻辑混杂在页面或控制器中。数据库实体与业务服务保持对应关系，有利于后续扩展。

在可测试性方面，系统应提供自动化测试覆盖核心流程。由于自动评测系统涉及异步判题、权限隔离和数据关联，如果缺少测试，后续修改容易破坏主流程。本系统通过集成测试验证登录、课程管理、作业创建、学生提交、编译错误、统计导出和 demo 数据初始化等功能。

在易演示性方面，系统应支持本地快速启动和固定演示数据。毕业设计答辩时间有限，如果每次演示都需要手动创建用户、课程和作业，容易出现操作失误。因此系统提供 demo profile，自动生成管理员、教师、学生、课程、选课关系和演示作业，保证主流程可以稳定复现。

## 3.5 系统用例分析

系统的主要用例可以分为三组。管理员用例包括登录系统、查看用户、查看课程、查看作业、查看操作日志和搜索分页数据。教师用例包括登录系统、创建课程、更新课程、管理选课、发布作业、配置测试用例、查看提交、重新判题、查看统计和导出成绩。学生用例包括登录系统、查看已选课程、查看已发布作业、提交代码、查看最新提交摘要和查看评测详情。

从用例之间的关系看，课程和选课是作业可见性的基础。只有当学生加入课程后，课程下已发布作业才对该学生可见。作业和测试用例是自动评测的基础，没有测试用例的作业无法给出有效的通过判断。提交记录是评测结果和统计分析的基础，系统需要保存每一次提交的源码、状态、分数和评测详情。各用例之间形成了较强的数据关联，这也是数据库设计和权限控制需要重点考虑的原因。

# 4 系统总体设计

## 4.1 系统架构设计

本系统采用单体 Web 应用架构，整体由前端静态页面、后端 REST 接口、业务服务层、数据访问层、数据库和自动评测模块组成。前端页面位于 `src/main/resources/static` 目录中，浏览器访问系统后加载 HTML、CSS 和 JavaScript 文件。前端通过 `fetch` 请求调用后端接口，并根据登录用户角色展示不同工作台。后端基于 Spring Boot 启动，Controller 层接收请求，Service 层处理业务逻辑，Repository 层访问数据库。

系统的数据流大致如下：用户在浏览器中登录，后端验证用户名和密码后生成 token；之后前端在请求头中携带 token 调用受保护接口；后端拦截器解析 token 并设置当前用户信息；Controller 调用对应 Service 完成业务处理；Service 通过 Repository 读写数据库；当学生提交代码时，提交服务创建 `Submission` 记录并提交异步评测任务；评测模块完成编译运行后更新提交状态和测试用例结果。

该架构的优点是层次清晰，部署简单，适合本科毕业设计项目。由于前端为静态资源，后端可以直接托管页面，不需要单独部署前端工程。由于系统规模适中，单体架构比微服务架构更容易开发和调试，也更适合答辩展示。后续如果需要支持更高并发或多语言评测，可以将自动评测模块拆分为独立服务。

## 4.2 功能模块设计

系统功能模块主要包括认证模块、用户管理模块、课程管理模块、作业管理模块、提交管理模块、自动评测模块、统计导出模块、操作日志模块和前端展示模块。认证模块负责注册、登录、登出和当前用户信息获取；用户管理模块负责用户列表、角色筛选、禁用启用和密码重置；课程管理模块负责课程创建、更新、删除、学生选课和课程统计；作业管理模块负责作业创建、更新、删除、测试用例增删改和作业查询；提交管理模块负责学生提交、提交查询、重新判题和最新摘要；自动评测模块负责源码编译、程序运行和结果计算；统计导出模块负责作业统计、课程统计和成绩导出；操作日志模块负责记录关键操作。

模块之间的关系体现了业务流程的先后顺序。用户认证模块是所有受保护功能的入口；课程模块为作业模块提供课程归属；作业模块为提交模块提供提交对象和测试用例；提交模块调用自动评测模块生成结果；统计模块基于作业、课程和提交数据生成分析结果；日志模块贯穿课程、作业、提交和用户管理等关键操作。通过模块化设计，系统各部分职责明确，便于维护。

## 4.3 数据库设计

数据库设计围绕教学业务对象展开。用户表 `users` 保存用户名、密码、姓名、班级、角色和启用状态。课程表 `courses` 保存课程代码、课程名称、学期、班级、是否启用和教师 ID。选课关系表 `course_enrollments` 保存课程与学生之间的多对多关系。作业表 `assignments` 保存作业标题、描述、截止时间、状态、评分策略、最大提交次数、是否允许迟交、教师 ID 和课程 ID。测试用例表 `test_cases` 保存输入数据、期望输出和所属作业。提交表 `submissions` 保存作业 ID、学生 ID、源码、主类名、状态、分数、编译信息、运行信息和提交时间。评测用例结果表 `judge_case_results` 保存每个测试用例的实际输出、是否通过和错误信息。登录令牌表 `auth_tokens` 保存 token 和过期时间。操作日志表 `audit_logs` 保存操作者、动作、对象和摘要。

这些表之间存在明确的关联关系。一个教师可以创建多门课程，一门课程可以关联多个学生选课记录；一个课程可以包含多个作业，一个作业可以包含多个测试用例和多个提交记录；一个提交记录可以包含多个评测用例结果。通过这些关系，系统能够追踪从课程到作业再到提交和评测详情的完整数据链路。

数据库设计还考虑了查询效率和完整性。课程代码和用户名具有唯一性，避免重复数据；作业表对教师 ID 和课程 ID 建立索引，便于教师按课程和作业查询；提交表对作业 ID、学生 ID 以及二者组合建立索引，便于查询某个作业的全部提交、某个学生的提交以及某个学生在某作业下的最新提交。对于毕业设计系统而言，该设计既能满足功能需求，也便于论文中通过 ER 图展示数据关系。

## 4.4 权限控制设计

系统采用基于 token 的认证机制。用户登录成功后，后端生成随机 token，并将 token、用户 ID、签发时间和过期时间保存到数据库。前端将 token 保存在浏览器本地存储中，后续请求通过 `Authorization: Bearer token` 方式携带。后端拦截器在请求进入 Controller 前解析 token，验证其是否存在、是否过期、对应用户是否启用，并将当前用户信息放入请求属性中。

权限控制分为认证和授权两层。认证用于判断用户是否登录，未登录或 token 无效时返回 `401`。授权用于判断用户角色和数据归属是否满足要求，角色不匹配或访问他人数据时返回 `403`。例如，学生不能创建作业，教师不能提交学生代码，学生不能查看其他学生提交，教师不能管理其他教师创建的课程和作业。系统在 Service 层集中进行关键权限判断，避免仅依赖前端隐藏按钮造成安全风险。

数据范围控制是本系统权限设计的重点。对于学生端作业查询，系统不仅要求作业状态为 `PUBLISHED`，还要求学生已选修该作业所属课程。对于教师端课程和作业管理，系统要求当前教师为课程或作业创建者，管理员则可以查看全局数据。通过这种设计，系统能够比较真实地模拟教学场景中的权限边界。

## 4.5 自动评测流程设计

自动评测流程从学生提交代码开始。学生选择作业并提交 Java 源码后，系统首先校验当前用户必须为学生，作业必须存在且处于已发布状态，学生必须已选修作业所属课程，提交次数和截止时间也需要符合要求。校验通过后，系统创建提交记录，初始状态为 `PENDING`，随后在事务提交后将提交 ID 放入异步评测任务。

评测任务读取提交记录和作业测试用例，调用 Java 评测服务。评测服务首先识别源码中的主类名，并检查源码长度和危险关键字；然后使用 `JavaCompiler` 编译源码。如果编译失败，系统返回编译错误并保存诊断信息。如果编译成功，系统依次执行测试用例，将测试输入写入程序标准输入，读取标准输出和错误输出。每个测试用例执行完成后，系统将实际输出与期望输出进行规范化比较，记录是否通过。

所有测试用例执行完成后，系统根据通过数量计算得分。若全部通过，则状态为 `ACCEPTED`；若部分通过，则状态为 `PARTIAL_ACCEPTED`；若全部未通过且无运行错误，则状态为 `FAILED`；若发生编译错误、运行错误或超时，则返回对应状态。最后，系统将评测状态、分数、编译信息、运行信息和每个测试用例结果保存到数据库中。学生和教师随后可以通过提交详情接口查看完整评测结果。

## 4.6 Demo 数据设计

为了提高系统演示的稳定性，本系统设计了 `demo` profile。使用 demo 配置启动时，系统使用 H2 内存数据库，并自动创建管理员、教师和学生演示账号。演示账号包括 `admin1/123456`、`teacher1/123456` 和 `student1/123456`。系统还会创建课程 `CS101 · 程序设计基础`，将 `student1` 加入该课程，并创建已发布作业 `演示：两数求和`。该作业配置两条测试用例：输入 `1 2` 期望输出 `3`，输入 `6 9` 期望输出 `15`。

Demo 数据不仅是初始化数据，也是答辩演示脚本的一部分。通过固定数据，答辩时可以直接展示管理员端、教师端和学生端功能，不需要现场临时录入课程和作业。为了防止演示数据被手动修改导致主流程失败，系统在 demo 启动时会将演示作业校正为可发布、可提交、可评测状态，并固定测试用例内容。这样可以保证学生端默认代码能够稳定得到 `ACCEPTED` 和 100 分，提高演示可靠性。

## 4.7 核心接口设计

系统接口按照资源类型划分，主要包括认证接口、用户接口、课程接口、作业接口、提交接口、导入接口、管理员接口和审计日志接口。认证接口 `/api/auth` 提供注册、登录、退出和当前用户查询功能，是系统使用流程的入口。用户接口 `/api/users` 主要用于教师或管理员查询用户信息，管理员还可以启用、禁用用户并重置密码。课程接口 `/api/courses` 用于课程创建、修改、删除、选课管理和课程统计。作业接口 `/api/assignments` 用于作业发布、作业查询、测试用例管理、作业统计和成绩导出。提交接口 `/api/submissions` 用于学生提交代码、查询提交详情、教师查看作业提交和重新判题。

接口设计遵循“前端只负责交互，后端负责权限和业务判断”的原则。比如学生端虽然只展示可提交作业，但后端仍会在提交接口中检查作业状态、课程选课关系、提交次数和截止时间。教师端虽然只能选择自己创建的课程，但后端在课程和作业服务中仍会检查资源归属。这样设计可以避免仅依赖前端控制带来的安全风险。接口返回结果统一使用 JSON 格式，错误响应中包含状态码、错误时间和错误消息，便于前端进行提示，也便于调试和测试。

核心接口的粒度保持适中。对于常用列表查询，例如学生查看已发布作业、教师查看作业提交、管理员分页搜索用户，系统提供专门接口，减少前端拼接复杂查询的负担。对于具有业务含义的操作，例如重新判题、成绩导出、选课和退课，系统使用明确的接口路径表达操作对象和动作含义。这样的接口组织方式能够提高系统可读性，也便于在论文和答辩中说明系统设计。

# 5 系统详细设计与实现

## 5.1 用户认证模块实现

用户认证模块主要由 `AuthController`、`UserService`、`PasswordService` 和 `AuthTokenService` 组成。用户登录时，前端将用户名和密码提交到 `/api/auth/login` 接口。后端根据用户名查询用户，如果用户不存在、密码错误或账号被禁用，则返回对应错误信息；如果验证成功，则生成认证会话并返回用户 ID、用户名、角色、token 和过期时间。

密码处理通过 `PasswordService` 完成，系统不会直接保存明文密码，而是保存编码后的密码。Token 由 `AuthTokenService` 生成并写入 `auth_tokens` 表，过期时间由配置项控制。用户退出登录时，系统删除当前 token，使其立即失效。当前用户信息接口 `/api/auth/me` 用于前端刷新页面后恢复登录状态。

认证拦截器是模块的重要组成部分。除登录、注册和静态资源外，受保护接口都需要经过拦截器校验。拦截器读取请求头中的 Bearer token，查询数据库确认 token 有效，并构造 `AuthenticatedUser` 对象传递给后续 Controller 和 Service。通过这种方式，业务层可以方便地获得当前用户 ID 和角色，从而进行权限判断。

## 5.2 用户与权限管理实现

用户管理模块主要用于管理员维护系统用户。管理员可以查看用户列表，按角色筛选用户，禁用或启用用户账号，也可以重置用户密码。当某个用户被禁用后，系统会使其已有 token 失效，并禁止其再次登录。该功能可以用于处理账号异常或教学管理中的人员变更。

权限管理并不是单独存在的页面功能，而是贯穿于各业务服务中。系统通过 `ServiceHelper.ensureTeacher` 和 `ServiceHelper.ensureAdmin` 等方法统一判断角色。教师相关接口要求当前用户为教师或管理员，管理员接口要求当前用户为管理员。学生提交代码时，系统要求当前用户必须为学生。对于涉及数据归属的操作，例如教师更新课程、删除课程、修改作业、查看提交，系统还会检查资源是否属于当前教师。

这种权限实现方式既保持了代码清晰，也提高了安全性。前端虽然会根据角色隐藏无权限功能，但真正的权限判断由后端完成。即使用户绕过页面直接调用接口，后端也会返回禁止访问错误。自动化测试中也覆盖了未登录、越权访问、学生访问教师资源等场景，验证权限控制的有效性。

## 5.3 课程与选课模块实现

课程与选课模块由 `CourseController` 和 `CourseService` 实现。教师可以创建课程，填写课程代码、课程名称、学期和班级。系统会对课程代码进行唯一性校验，避免重复课程造成管理混乱。教师还可以修改课程信息、删除无作业的课程以及查看课程统计信息。

选课管理用于建立课程与学生之间的关系。教师在课程管理页面选择课程后，可以将学生加入课程，也可以将已选学生移出课程。后端通过 `course_enrollments` 表保存课程 ID、学生 ID 和选课时间，并对同一课程和同一学生建立唯一约束，避免重复选课。只有学生角色用户可以被加入课程，如果目标用户不是学生，系统会返回错误。

学生端课程可见性是该模块的关键。学生调用 `/api/courses` 接口时，系统不会返回所有课程，而是通过选课关系查询当前学生已加入且启用的课程。教师调用同一接口时，系统返回教师自己创建的课程；管理员可以查看全部课程。这样，一个接口根据角色返回不同范围的数据，既简化了前端调用，也保证了权限隔离。

## 5.4 作业与测试用例模块实现

作业管理模块由 `AssignmentController` 和 `AssignmentService` 实现。教师创建作业时，需要填写作业标题、描述、截止时间、所属课程、状态、最大提交次数、是否允许迟交和评分策略。系统支持 `DRAFT`、`PUBLISHED` 和 `CLOSED` 三种作业状态。草稿作业不会对学生开放，已发布作业可以提交，关闭作业不能继续提交。

测试用例是自动评测的依据。教师可以在创建作业时同时提交测试用例，也可以在作业创建后新增、修改和删除测试用例。每个测试用例包括输入数据和期望输出。评测时系统会按照测试用例顺序运行程序，并保存每个用例的实际输出和通过情况。对于学生端作业查询，系统会隐藏测试用例答案，避免学生直接看到期望输出导致评测失去意义。

作业模块还支持评分策略和提交限制。`LATEST` 策略表示以最后一次提交作为有效成绩，`HIGHEST` 策略表示以最高分提交作为有效成绩。最大提交次数限制可以防止学生无限提交。是否允许迟交用于处理截止时间后的提交规则。通过这些字段，系统能够覆盖常见课程作业管理需求。

## 5.5 学生代码提交模块实现

学生提交模块由 `SubmissionController` 和 `SubmissionService` 实现。学生在前端选择作业后，填写或修改 Java 源码并点击提交。前端将作业 ID 和源码发送到 `/api/submissions` 接口。后端首先确认当前用户为学生，然后加载作业并检查作业状态、截止时间、提交次数和课程选课关系。只有满足所有条件时，系统才会创建提交记录。

提交记录创建后，状态初始为 `PENDING`。为了避免评测过程阻塞 HTTP 请求，系统在事务提交后触发异步评测任务。前端提交成功后会提示“已加入后台评测队列”，并通过轮询方式刷新提交详情。当评测完成后，页面会显示最终状态、分数、编译信息、运行信息和测试用例结果。

提交模块还提供教师和学生不同范围的查询能力。学生只能查看自己的提交记录和最新摘要，教师可以查看自己作业下的学生提交，管理员可以进行全局查看。教师还可以对提交执行重新判题，用于测试用例修改后的结果刷新。重新判题会将提交状态重新设置为 `PENDING`，并再次进入评测队列。

## 5.6 Java 自动评测模块实现

Java 自动评测模块由 `JavaJudgeService` 和 `SubmissionJudgeQueueService` 共同完成。`SubmissionJudgeQueueService` 负责异步读取提交记录、调用评测服务并保存结果；`JavaJudgeService` 负责具体的编译和运行逻辑。系统首先从源码中识别 public class 名称，如果未识别到，则默认使用 `Main`。随后进行源码校验，包括长度限制和部分危险关键字限制。

编译阶段使用 JDK 提供的 `ToolProvider.getSystemJavaCompiler()` 获取 Java 编译器。系统将学生源码写入临时目录中的 `.java` 文件，然后调用编译器进行编译。编译失败时，系统收集诊断信息并返回 `COMPILE_ERROR`。编译成功后，系统根据测试用例运行程序。每次运行都会创建独立进程，并将测试输入写入标准输入。系统读取标准输出和标准错误输出，并控制最大执行时间和最大输出长度。

结果判断阶段，系统会对实际输出和期望输出进行规范化处理，减少换行差异对结果的影响。如果两者一致，则该测试用例通过；否则不通过。若某个用例发生超时或运行错误，系统记录错误信息并避免后续用例继续造成资源浪费。最终，系统根据通过用例数量计算百分制分数，并设置整体状态。该实现方式逻辑清晰，适合展示自动评测的核心原理。

## 5.7 结果查询与统计模块实现

结果查询模块为教师和学生提供不同视角。学生可以查看自己的所有提交，也可以查看每个作业的最新提交摘要。提交详情中包含作业标题、学生信息、源码、状态、分数、编译信息、运行信息和测试用例结果。教师可以查看某个作业下所有学生提交，并查看每条提交的详情。这样，学生能够根据反馈修改程序，教师也能了解学生完成情况。

统计模块主要包括作业统计、课程统计和成绩导出。作业统计会计算提交次数、参与学生数和平均分。课程统计会汇总课程下的选课人数、作业数量、已发布作业数量、提交总数和平均成绩。成绩导出功能以 CSV 形式导出某作业下学生成绩，内容包括用户名、姓名、班级、状态、成绩、提交时间、提交次数和有效提交 ID。

统计模块在教学场景中具有实际意义。教师不仅需要查看某一次提交是否通过，也需要掌握整体完成情况。例如，如果某个作业平均分较低，说明题目难度较高或相关知识点掌握不足；如果提交人数较少，教师可以提醒学生及时完成。通过统计和导出功能，系统能够从“自动判题工具”进一步扩展为“教学管理辅助系统”。

## 5.8 前端交互页面实现

系统前端采用静态 HTML、CSS 和 JavaScript 实现，由 Spring Boot 直接托管。页面根据用户角色展示管理员、教师和学生三个工作台。管理员端提供用户、课程、作业和日志列表；教师端提供课程与选课、作业创建、作业管理、提交查看、评分统计和批量导入等功能；学生端提供课程选择、已发布作业、代码提交、最近提交和评测详情等功能。

前端在登录成功后保存 token，并在后续请求中自动添加认证头。页面加载时会调用当前用户接口恢复会话，根据角色自动跳转到对应工作台。为了提高演示效率，登录页提供演示账号填充功能，用户可以快速填写 `admin1`、`teacher1` 或 `student1` 账号。

学生端交互是本系统答辩演示的重点。学生登录后可以看到已选课程 `CS101`，再进入已发布作业列表选择 `演示：两数求和`。代码提交区默认填充一段两数求和 Java 程序，提交后系统自动评测并展示结果。这样的设计可以在较短时间内完整展示课程、作业、提交和评测流程。

## 5.9 Demo 配置与演示链路实现

为了降低本地部署和答辩演示的不确定性，系统专门实现了 demo 配置。demo 配置文件将数据库切换为 H2 内存数据库，并关闭 Flyway 迁移，启动时由 JPA 自动创建表结构。`DataSeeder` 类只在 demo profile 下生效，它会在应用启动后写入演示账号、课程、选课关系和演示作业。与普通初始化数据不同，该类会对演示作业进行校正，确保其标题、描述、截止时间、状态、所属课程、提交次数和测试用例均处于可演示状态。

这种设计解决了毕业设计演示中常见的两个问题。第一，避免环境依赖过重。评阅教师或答辩现场不一定方便配置 MySQL，因此 H2 内存数据库可以快速启动系统。第二，避免现场录入数据导致流程中断。系统启动后即可使用固定账号登录，并直接展示完整业务链路。管理员账号用于展示后台管理能力，教师账号用于展示课程、选课和作业配置，学生账号用于展示课程可见性、作业提交和评测结果。

README 中也给出了对应的演示脚本。实际答辩时，可以先用管理员查看用户、课程、作业和日志，再用教师查看 `CS101` 课程和演示作业，最后用学生提交默认两数求和代码并查看 `ACCEPTED` 结果。该演示链路覆盖了系统最核心的设计目标，即从课程管理到作业发布，再到学生提交和自动评测的完整闭环。

# 6 系统测试

## 6.1 测试环境

系统测试环境包括 Windows 操作系统、JDK 17 及以上版本、Maven 3.9 及以上版本、Spring Boot 3.3、H2 内存数据库和浏览器。正式运行时系统可连接 MySQL 数据库，自动化测试和答辩演示主要使用 H2 数据库。使用 H2 的优势是启动快速、无需外部数据库配置，并且每次测试都可以使用干净的数据环境。

系统测试命令为 `mvn test`。Maven 会自动编译项目并执行 JUnit 测试。由于部分测试涉及 Spring Boot 上下文和 MockMvc 接口调用，测试过程能够覆盖从 Controller 到 Service 再到 Repository 的完整链路。自动评测相关测试还会实际编译和运行 Java 代码，从而验证评测模块的有效性。

## 6.2 功能测试

功能测试主要验证系统是否按照需求完成业务操作。认证测试包括用户注册、登录、退出和当前用户查询。课程测试包括教师创建课程、更新课程、删除无作业课程、禁止删除已有作业课程以及学生只能看到已选课程。作业测试包括教师创建作业、学生查看已发布作业、学生端测试用例脱敏和教师查看完整作业信息。提交测试包括学生提交正确代码、教师查看提交和学生查看最新摘要。

在手工演示测试中，使用 demo profile 启动系统后，可以按照管理员、教师、学生的顺序执行主流程。管理员登录后查看系统数据；教师登录后查看 `CS101` 课程、已选学生和演示作业；学生登录后查看课程和作业，提交默认 Java 代码；系统评测完成后显示 `ACCEPTED` 和 100 分；教师再登录查看该作业下的提交记录。该流程验证了系统主要功能的完整性。

## 6.3 权限测试

权限测试用于验证系统能否正确阻止非法访问。未携带 token 的请求应返回未认证错误，登录过期或 token 无效也应返回未认证错误。学生访问教师接口、教师访问管理员接口、教师提交学生代码、学生查看他人提交等行为应返回禁止访问错误。系统测试中对这些场景进行了覆盖，确保权限控制不仅依赖前端页面隐藏，而是在后端业务层真正生效。

数据归属测试也是权限测试的重要组成部分。教师只能管理自己创建的课程和作业，不能查看其他教师作业下的提交记录；学生只能查看自己已选课程下的已发布作业，不能提交未选课程作业。通过这些测试，可以证明系统在多用户场景下具有基本的数据隔离能力。

## 6.4 自动评测测试

自动评测测试覆盖多种程序运行结果。对于正确代码，系统应返回 `ACCEPTED`，分数为 100，所有测试用例通过。对于编译错误代码，系统应返回 `COMPILE_ERROR`，并保存编译错误信息。对于错误答案，系统应返回 `FAILED` 或 `PARTIAL_ACCEPTED`，分数根据通过用例比例计算。对于运行异常或死循环代码，系统应返回运行错误或超时状态。

系统还测试了重新判题功能。当教师修改测试用例后，可以对已有提交重新判题，系统会清空原评测结果并重新执行评测。测试表明，重新判题后提交状态和分数能够根据新的测试用例变化。该功能体现了评测结果与测试用例配置之间的动态关系，也符合实际教学中修改评测标准后的需求。

## 6.5 集成测试结果

本系统使用 Maven 执行自动化测试，测试范围覆盖用户认证、权限控制、课程管理、选课管理、作业管理、代码提交、自动评测、重新判题、统计导出、批量导入、管理员分页搜索、demo 数据初始化和前端静态结构等核心功能。测试命令为 `mvn test`。

测试结果显示，系统共执行 35 个测试用例，失败数为 0，错误数为 0，跳过数为 0，构建结果为 `BUILD SUCCESS`。该结果表明系统核心业务流程能够正常运行，主要权限规则和评测逻辑能够通过自动化测试验证。对于本科毕业设计项目而言，自动化测试结果可以作为系统稳定性的重要支撑材料。

## 6.6 测试结论

综合功能测试、权限测试和自动评测测试结果可知，本系统能够满足作业自动评测与管理的主要需求。系统能够完成教师发布作业、学生提交代码、自动评测、结果查询和教师统计等核心流程；能够对未登录和越权请求进行控制；能够识别常见评测结果；能够通过 demo profile 快速复现答辩演示场景。虽然系统评测沙箱仍较轻量，但对于课程演示和本科毕业设计交付而言，已经具备较好的完整性和可用性。

# 结束语

本文设计并实现了一套基于 Spring Boot 的作业自动评测与管理系统。系统面向程序设计课程教学场景，围绕管理员、教师和学生三类角色建立了课程、选课、作业、测试用例、提交、评测结果和统计分析等功能。通过该系统，教师可以发布作业并配置测试用例，学生可以在线提交 Java 代码，系统能够自动完成编译、运行、输出比对和评分，教师和学生可以分别查看对应范围内的提交结果。

从系统实现角度看，本文完成了后端 REST 接口、数据库实体、业务服务、权限控制、自动评测模块和静态前端页面的设计与实现。系统使用 Spring Boot 简化项目搭建，使用 Spring Data JPA 完成数据持久化，使用 H2 提供演示环境，使用 JavaCompiler 和进程执行机制实现 Java 程序评测。通过自动化测试，系统验证了认证、权限、课程、作业、提交、评测和统计等核心业务流程。

本系统仍存在一些不足。首先，当前自动评测模块采用轻量级限制方式控制用户程序，尚未达到生产级沙箱安全标准。其次，系统目前主要支持 Java 语言，暂未扩展到 C、C++、Python 等多语言评测。再次，系统前端为静态页面，虽然能够满足演示需求，但在组件化、交互细节和可视化统计方面仍有提升空间。后续可以引入 Docker 沙箱、多语言评测、代码相似度检测、班级成绩可视化、题库管理和更完善的教学分析功能，使系统更加接近真实教学平台。
