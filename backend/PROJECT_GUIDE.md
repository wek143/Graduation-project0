# 毕业设计项目推进指南

本文档面向当前仓库的真实状态，目的是帮助你把“作业自动批改系统”从可运行代码推进到可交付毕业设计。

## 1. 项目目标

本项目的最终目标不是只完成几个接口，而是形成一套完整的毕业设计交付物，包括：

- 可运行的后端系统
- 可演示的教师端与学生端流程
- 清晰的数据库设计
- 可说明的自动判题逻辑
- 可复现的测试结果
- 可用于论文与答辩的图表、截图和材料

## 2. 当前项目现状

结合当前代码仓库，已经完成的内容主要包括：

- Spring Boot 单体项目结构已经搭建完成
- 用户、课程、作业、测试用例、提交记录、判题结果、审计日志等核心实体已经具备
- 鉴权机制已经打通，支持登录、登出、当前用户身份获取
- 教师、学生、管理员三类角色的基本权限控制已经存在
- 课程管理、作业管理、提交管理、批量导入等接口已经实现
- 内置静态前端页面已经具备基础可用性
- Java 判题链路已经可用，能够处理通过、编译错误、运行错误、超时等结果
- Demo 配置可直接使用 H2 启动
- 已补充关键自动化测试，覆盖鉴权、判题和核心提交流程

也就是说，这个项目已经不是“从零开始”，而是已经进入“稳定性、收口、答辩包装”阶段。

## 3. 当前仍需完善的重点

虽然主体功能已经具备，但距离高质量交付还差几部分：

### 3.1 工程质量收口

- 继续补核心回归测试，尤其是课程、选课、批量导入、统计接口
- 统一剩余中文文案、错误消息和提示信息
- 压缩无关调试输出，保证演示时终端和页面更干净
- 再走一轮端到端手工验证，确认没有角色权限和边界条件遗漏

### 3.2 演示流程优化

- 让管理员、教师、学生三种角色都能完成完整操作路径
- 准备固定演示数据，避免答辩时现场临时录入
- 固定演示顺序，保证每次展示结果一致

### 3.3 文档与论文材料

- 整理系统架构图
- 整理数据库 ER 图
- 整理模块划分图
- 整理关键流程图
- 整理测试结果截图与功能截图
- 整理判题算法说明

## 4. 推荐推进阶段

建议按下面四个阶段推进，不要在未稳定时继续堆新功能。

### 第一阶段：稳定核心链路

目标：

- 确保演示主路径稳定

优先检查内容：

- 教师登录
- 创建课程
- 学生选课
- 教师创建作业
- 学生查看已发布作业
- 学生提交代码
- 系统返回判题结果
- 教师查看提交与统计

完成标准：

- 核心主流程可以连续演示
- 正确代码能返回 `ACCEPTED`
- 编译错误代码能返回 `COMPILE_ERROR`
- 未登录和越权访问能返回合理错误

### 第二阶段：补齐自动化测试

目标：

- 把关键逻辑固化成回归测试，避免改一个地方坏一片

建议优先补的测试：

- 课程创建与选课流程测试
- 批量导入用户、课程、选课测试
- 作业更新、删除、测试用例增删改测试
- 最新提交、最新摘要、统计接口测试
- 边界条件测试：超时、无测试用例、已截止、超提交次数

完成标准：

- `mvn clean test` 能覆盖主要业务链路
- 常见回归问题能通过测试快速发现

### 第三阶段：整理交付文档

目标：

- 让项目不仅能跑，还能被老师快速理解和复现

建议补齐的文档：

- 中文 README
- 项目推进指南
- 测试说明
- 演示步骤说明
- 环境配置说明

完成标准：

- 第三方拿到仓库后可根据文档启动和演示
- 不需要现场解释太多环境细节

### 第四阶段：准备毕业论文与答辩材料

目标：

- 让代码成果能自然映射到论文结构和答辩展示

建议准备的内容：

- 课题背景与意义
- 功能需求分析
- 系统架构设计
- 数据库设计
- 关键模块设计
- 判题流程设计
- 测试与结果分析
- 项目总结与展望

完成标准：

- 论文中的每个核心模块，都能在系统中找到对应实现
- 答辩时可以从“设计 -> 实现 -> 演示 -> 测试”完整闭环讲清楚

## 5. 当前系统建议展示的功能点

答辩时建议重点展示以下内容，不要平均用力：

### 5.1 用户与权限

- 不同角色登录后看到不同工作台
- 教师和学生不能访问彼此权限接口

### 5.2 课程与作业管理

- 教师创建课程
- 学生选课
- 教师创建作业并配置测试用例

### 5.3 自动判题

- 学生提交正确代码，返回通过和分数
- 学生提交错误代码，返回编译错误或运行错误
- 查看测试用例明细

### 5.4 结果追踪

- 教师查看某作业下的提交列表
- 学生查看自己的最新提交摘要
- 教师查看作业统计信息

## 6. 推荐数据库表

结合当前实体设计，系统核心表主要包括：

- `users`
- `courses`
- `course_enrollments`
- `assignments`
- `test_cases`
- `submissions`
- `judge_case_results`
- `auth_tokens`
- `audit_logs`

这些表已经基本覆盖当前毕业设计系统的主要业务场景。

## 7. 推荐近期任务顺序

如果按“尽快交付”的思路推进，建议接下来这样做：

1. 继续补课程、选课、导入、统计的自动化测试
2. 清理剩余日志噪音和前后端提示文案
3. 固化演示数据与演示脚本
4. 输出系统架构图、ER 图、流程图
5. 整理论文对应章节提纲

## 8. 我们后续协作方式

推荐按以下方式继续推进：

1. 一次只推进一个明确目标
2. 每做完一批改动就立即验证
3. 先保证可用，再考虑扩展
4. 以“可交付”和“可答辩”为标准，而不是一味继续加功能

## 9. 当前最推荐的下一步

当前最合适继续推进的方向是：

`继续补课程、选课、导入、统计相关的自动化测试，并同步整理答辩演示材料。`

原因很直接：

- 主链路已经基本打通
- 现在最需要的是稳定性和展示质量
- 这些工作最能直接提升交付完成度
